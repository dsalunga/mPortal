Status legend:
- `Completed`: modern target (`net8+`) is declared in project metadata.
- `Partial`: project has evidence of split legacy/modern coexistence.
- `Not Stated`: no explicit migration status marker found.
- `Blocked`: migration blocked by dependency/constraint (to be filled by owners).
- `Do Not Migrate As-Is`: retain/replace/retire; do not lift-and-shift this artifact.
- `Not Started`: explicitly tracked but not yet begun (to be filled by owners).
- `Deprecated`: retained as legacy/deprecated artifact.

## Canonical Migration Decision Matrix

This matrix maps the canonical legacy card set in [../canonical/legacy/](../canonical/legacy/) (cards `00` through `19`) to migration decisions so implementation stays focused on core/essential outcomes.

Decision status legend:
- `Core - Migrate`: required in target architecture.
- `Core - Migrate With Redesign`: required outcome, but not a direct lift-and-shift.
- `Selective - Validate First`: migrate only validated/high-usage subsets.
- `Retain Legacy Temporarily`: keep in legacy during phased cutover.
- `Do Not Migrate As-Is`: replace, retire, or reimplement differently.

### Do Not Migrate As-Is (Validated)

| Area | Decision | Why |
|---|---|---|
| Vendored sample/demo assets (e.g., legacy plugin docs/samples) | Do Not Migrate As-Is | Not business functionality; replace via package-managed dependencies |
| NAnt + junction-specific release orchestration | Do Not Migrate As-Is | Environment-bound and non-reproducible; replace with modern CI/CD (GitHub Actions/Azure DevOps) |
| Utility GUIs with unclear ownership/use (example: legacy table editors) | Do Not Migrate As-Is | High maintenance risk, unclear product value; replace with controlled internal web/CLI tools with RBAC and audit logs |

### Cannot Directly Migrate (Needs Redesign)

| Area | Constraint | Target Direction |
|---|---|---|
| Legacy membership/session/auth patterns | Old framework assumptions and weak secret handling | Modern identity provider (OIDC + token/session model), secrets vault, and password rehash migration path |
| ASMX/WCF mixed integration endpoints | Transport/runtime mismatch with modern service posture | Strangler pattern with REST/JSON API adapters plus endpoint-by-endpoint client migration |
| Reflection-heavy provider wiring and global state | Implicit behavior is hard to reason about/test | Explicit dependency injection + typed service boundaries |
| Thread-based agent execution with abort semantics | Unsafe cancellation and limited reliability guarantees | Durable job runners (Hangfire/Quartz.NET/.NET Worker) with retries, idempotency, DLQ/error handling, and observability |

### Minimum Core Feature Set To Protect

1. Security + identity + authorization semantics (`05`).
2. Core domain object integrity and metadata behavior (`03`, `04`).
3. Request routing/rendering behavior for active user journeys (`02`, `07`).
4. Business-critical module workflows from G1/G2/G3/Integration (`08`-`11`).
5. Reliable data migration/recovery and release pipeline (`14`, `15`).

### Explicit Alternatives Checklist

1. Legacy editor package (`FCKeditor`): options considered were `CKEditor 5`, `TinyMCE`, and `TipTap`; selected target is `TipTap OSS` plus server-side HTML sanitization for all migrated editor usage.
2. ASMX/WCF/ASHX endpoints: use a strangler pattern with API adapters (`REST/JSON`) in front, then migrate clients endpoint-by-endpoint.
3. Thread-abort agent/service jobs: replatform to durable jobs (`Hangfire`, `Quartz.NET`, or worker services) with retries, idempotency, and DLQ/error handling.
4. NAnt + junction script release flow: replace with CI/CD pipelines (`GitHub Actions`/`Azure DevOps`), deterministic builds, and artifact promotion.
5. Desktop deploy/DB utilities: replace with controlled internal web/CLI ops tools with RBAC and audit logs.
6. Legacy auth/session/security flows: migrate to modern identity (`OIDC` + token/session model), secrets vault, and password rehash migration path.
7. Vendored sample/demo assets (bootstrap/fckeditor samples/docs): exclude from migration scope; keep only business-used assets.
8. Legacy SQL/script restore model: move to migration-based schema/versioning (`Flyway`/`Liquibase`/`EF migrations`) plus tested rollback strategy.
